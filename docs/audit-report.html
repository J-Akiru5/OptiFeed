OptiFeed Firmware-Backend Integration Audit

Security, reliability, and data-integrity assessment of the ESP32-to-Next.js data pipeline

Date: July 11, 2026
Scope: /api/ingest, /api/feed-command, Prisma schema, firmware
Deployment: Vercel serverless + Supabase PostgreSQL
Scale: 1 device, 1 farmer

Executive Summary

This audit examined the data pipeline between the OptiFeed ESP32 firmware and the Next.js backend, focusing on data integrity, reliability, and security at the current scale (one device, one farmer). Seven issues were identified across event ingestion, request lifecycle, and offline detection. All seven have been remediated in this pass.

#1 — Idempotent Event Ingestion Critical Fixed
Risk
Silently duplicated feed events corrupt FCR math
Location
apps/web/app/api/ingest/route.ts, prisma/schema.prisma, firmware/firmware-main.ino

Only dashboard-triggered feeds carried a feed_request_id. Scheduled and button-triggered events had no dedup key at all. If the ESP32's HTTP client times out waiting for a response the backend actually received, it retries and creates a duplicate feed_dispensed row, which quietly corrupts FCR calculations.

Before — No event ID (ingest/route.ts)
// No unique identifier on the event
const feedDispensedSchema = baseSchema.extend({
  grams: z.number().positive(),
  source: z.enum(["scheduled", "dashboard", "button"]),
  feed_request_id: z.string().nullable().optional(),
  // No event_id field — retries create duplicate rows
});
After — Event ID with unique constraint + idempotent retry
// Firmware generates a unique ID per event (esp_random + counter)
String generateEventId() {
  eventCounter++;
  uint32_t r = esp_random();
  char buf[24];
  snprintf(buf, sizeof(buf, "evt-%08lx-%04lx", r, eventCounter & 0xFFFF);
  return String(buf);
}

// Backend schema: z.string().min(8) — not .uuid() — no RFC4122 requirement
const baseSchema = z.object({
  event_id: z.string().min(8).optional(),  // any unique string, not strict UUID
  // ...
});

// Prisma schema: @unique constraint enables dedup at the DB level
model FeedEvent {
  eventId  String?  @unique
  // ...
}

// Ingest route: catch P2002 (unique constraint) → return 204 (idempotent success)
try {
  await prisma.$transaction([...]);
} catch (error) {
  if (isUniqueConstraintError(error)) {
    return new Response(null, { status: 204 }); // already received
  }
  throw error;
}

Design decision: The backend uses z.string().min(8) instead of z.string().uuid() because esp_random() does not produce RFC4122-compliant UUIDs. The firmware generates evt-<hex>-<counter> strings, which are unique without requiring spec compliance. Forcing the ESP32 to format a perfect UUID just to satisfy a validator that only needs uniqueness would be unnecessary complexity.

#2 — Server-Side Received Timestamp High Fixed
Risk
Device clock drift makes event ordering unreliable
Location
prisma/schema.prisma — FeedEvent model

The ESP32's clock is self-reported and can be wrong (RTC not synced, drifted, battery dead). The backend was storing the device's timestamp as the only time field, with no server-side record of when the event was actually received. This makes event ordering unreliable and hides devices with broken clocks.

Before — Only device-reported timestamp
model FeedEvent {
  timestamp   String    // device-reported, may be "unknown" or wrong
  createdAt   DateTime  @default(now())  // unnamed ambiguously
  @@index([deviceId, createdAt])
}
After — Renamed to receivedAt, kept device timestamp separate
model FeedEvent {
  timestamp   String    // device-reported (kept for forensics)
  receivedAt  DateTime  @default(now())  // server-stamped, trusted for ordering
  @@index([deviceId, receivedAt])
}

Migration safety: The column rename uses ALTER TABLE ... RENAME COLUMN via a hand-written migration SQL, not Prisma's default migrate dev which commonly generates DROP + ADD (wiping existing data). The migration was created with --create-only and the SQL was verified to use RENAME COLUMN.

#3 — Zod safeParse Crash High Fixed
Risk
Malformed payload crashes the entire route with HTTP 500
Location
apps/web/app/api/ingest/route.ts:63, :85

heartbeatSchema.parse(body) and feedDispensedSchema.parse(body) were called with .parse() instead of .safeParse(). On invalid input, .parse() throws a ZodError, which is unhandled — the route crashes with a 500 instead of returning a clean 422.

Before — .parse() throws on invalid input
const parsed = heartbeatSchema.parse(body);  // throws ZodError → 500
// ...
const parsed = feedDispensedSchema.parse(body);  // throws ZodError → 500
After — .safeParse() returns a result object
const parsed = heartbeatSchema.safeParse(body);
if (!parsed.success) {
  return new Response("Unprocessable Entity", { status: 422 });
}
// parsed.data is now type-safe
// ...
const parsed = feedDispensedSchema.safeParse(body);
if (!parsed.success) {
  return new Response("Unprocessable Entity", { status: 422 });
}
#4 — Missing Database Indexes Medium Fixed
Risk
Slow History page as data grows; not urgent at prototype volume
Location
prisma/schema.prisma — BiomassLog, FeedEvent

FeedEvent had an index on (deviceId, createdAt) but that column was renamed to receivedAt (Finding #2) — the index needed updating. BiomassLog had no index on (pondId, recordedAt), which the History page queries against.

After — Both indexes added
model FeedEvent {
  // ...
  @@index([deviceId, receivedAt])  // updated from createdAt
}

model BiomassLog {
  // ...
  @@index([pondId, recordedAt])    // new — for History page queries
}
#5 — Pending-Request Expiry (Lazy) High Fixed
Risk
Stale "pending" requests block new feed commands forever
Location
apps/web/app/api/feed-command/route.ts

If a dashboard "Feed Now" request never gets acknowledged (device offline, goes to sleep, etc.), the FeedRequest stayed "pending" forever. The next time the farmer taps "Feed Now", the double-tap guard (Finding #7) would block it because a pending request already exists.

After — Lazy expiry checked on every ESP32 poll
// No cron job — checked lazily when the device polls
const REQUEST_EXPIRY_MS = 60_000;  // 60 seconds

// Mark any pending requests older than 60s as expired
await prisma.feedRequest.updateMany({
  where: {
    deviceId: device.id,
    status: "pending",
    createdAt: { lt: new Date(Date.now() - REQUEST_EXPIRY_MS) },
  },
  data: { status: "expired" },
});

// Then query for genuinely pending requests
const pendingRequest = await prisma.feedRequest.findFirst({
  where: { deviceId: device.id, status: "pending" },
  orderBy: { createdAt: "asc" },
});
#6 — Device-Offline Detection (Lazy) Critical Fixed
Risk
Dashboard shows "online" for a device that hasn't reported in hours
Location
apps/web/app/[locale]/(dashboard)/dashboard/page.tsx, components/EnergyControllerCard.tsx

The dashboard's critical alert banner was hardcoded to device.connectivity === "offline" — a static string on the mock Device model, not derived from actual ESP32 telemetry. A device that stopped reporting heartbeats would still show as "online" on the dashboard.

Before — Hardcoded connectivity string
// Dashboard page
const isOffline = device.connectivity === "offline";
const criticalAlert = isOffline;

// EnergyControllerCard — no offline prop, no lastSeenAt-based logic
After — Lazy offline detection from lastSeenAt
// Dashboard page — compute from actual ESP32 last heartbeat
const OFFLINE_THRESHOLD_MS = 15 * 60 * 1000;  // 15 minutes
const esp32Offline = !energyDevice?.lastSeenAt ||
  (now.getTime() - energyDevice.lastSeenAt.getTime() > OFFLINE_THRESHOLD_MS);
const criticalAlert = esp32Offline && !!energyDevice;

// EnergyControllerCard — receives isOffline, shows red badge + warning banner
{isOffline && (
  <div className="bg-red-50 border-red-200">
    Device offline — last seen {formatLastSeen(lastSeenAt)}.
    Scheduled and button feeds still work; dashboard commands will queue.
  </div>
)}

Lazy approach: No background job or cron is needed. The offline check runs when the dashboard page is rendered — it compares lastSeenAt to the current time. For one device, a scheduled function would be solving a scale problem that doesn't exist yet.

#7 — Double-Tap Guard (Server-Side) Medium Fixed
Risk
Rapid tapping creates multiple pending feed requests
Location
apps/web/lib/actions/energy.ts — requestFeed()

The "Feed Now" button was disabled client-side during the server action (isPending), but after the action completed, a 3-second visual feedback window was the only protection. requestFeed() created a new FeedRequest unconditionally every call — no server-side check for an already-pending request.

Before — Creates a new request every time
export async function requestFeed(deviceId: string, grams: number) {
  await prisma.feedRequest.create({
    data: { deviceId, grams, status: "pending" },
  });
  revalidatePath("/[locale]/(dashboard)/dashboard", "page");
  return { success: true };
}
After — Checks for existing pending request first
export async function requestFeed(deviceId: string, grams: number) {
  const existing = await prisma.feedRequest.findFirst({
    where: { deviceId, status: "pending" },
  });

  if (existing) {
    return { success: true, message: "Feed already pending" };
  }

  await prisma.feedRequest.create({
    data: { deviceId, grams, status: "pending" },
  });
  revalidatePath("/[locale]/(dashboard)/dashboard", "page");
  return { success: true };
}

Data Flow Diagram

The complete data pipeline after all fixes are applied:

ESP32 (every 5s) ├── POST /api/ingest → event_id (unique) + heartbeat or feed_dispensed │ Auth: X-Device-Token header │ → Prisma: update EnergyDevice.lastSeenAt, create FeedEvent │ → If duplicate event_id: return 204 (idempotent) │ └── GET /api/feed-command → lazy-expire pending > 60s, then check for new requests Auth: X-Device-Token header → Returns: { feed_requested, feed_request_id, grams, grams_per_feeding } Dashboard UI ├── "Feed Now" button → requestFeed() server action │ → Check for existing pending request (double-tap guard) │ → Prisma: create FeedRequest (status: "pending") │ └── Dashboard page render → Compute isOffline from lastSeenAt (15min threshold) → Pass isOffline to EnergyControllerCard → Card shows red badge + warning banner if offline Feed Request Lifecycle pending → dispatched (ESP32 polled) → completed (feed_dispensed event received) pending → expired (60s timeout, lazy-checked on next poll)

Deliberately Skipped (Proportionate to Scale)

These items were considered but explicitly skipped to avoid over-engineering for a one-device, one-farmer system:

Per-device token rotation/revocation — One device, one token. Revisit if this becomes multi-farm.

Rate limiting beyond basic token validation — 5-second polling from one device is ~17k requests/day, trivial load. Add when the URL is more discoverable.

Dedicated cron/scheduler service — Offline-detection and request-expiry are handled lazily at render/poll time. No background jobs needed.

Observability tooling (Sentry, Datadog) — Vercel's built-in function logs are sufficient at this scale. Add when there's an incident history.

mTLS or signed-request schemes — HTTPS + a static bearer token is proportionate for one device. The threat model doesn't justify the complexity.

Naming debt flagged but not fixed: The Prisma models are still named EnergyDevice / EnergyReading (carried over from the original energy-monitoring template). Now is the cheapest time to rename them (e.g. FeederDevice / FeedEvent), while only two developers touch this code. Once more screens and migrations are built against these names, the cost increases permanently. This was outside this audit's scope but worth a deliberate decision.

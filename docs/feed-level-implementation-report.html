System Overview

OptiFeed is a full-stack IoT shrimp feeding management platform. An ESP32 microcontroller runs an automatic feeder with 3-way fallback (RTC schedule, remote dashboard commands, physical button). The web app — Next.js 16 + Supabase/Postgres + Prisma — provides real-time monitoring, biomass tracking, and bilingual UI (English + Hiligaynon). The newly added Feed Level Monitoring feature integrates an HC-SR04 ultrasonic sensor into the existing hardware, telemetry pipeline, database, and dashboard.

1. Database Schema CHANGED

EnergyDevice — 6 new fields

prisma/schema.prisma → model EnergyDevice
feedLevelPercent Float? — 0-100%, null until first ultrasonic reading
feedLevelCm Float? — raw distance in cm (debugging/recalibration)
feedLevelUpdatedAt DateTime? — last heartbeat with a reading
hopperEmptyCm Float? — sensor distance when hopper is empty
hopperFullCm Float? — sensor distance when hopper is full
hopperCapacityG Float? — total capacity in grams (e.g. 7000 for 10L)

FeedLevelLog — new model

prisma/schema.prisma → model FeedLevelLog
Historical log for trend charts (Growth page). Indexed on [deviceId, recordedAt].
Logged sparingly: only when 30min have passed OR level shifts >5%.

2. ESP32 Firmware CHANGED

File: firmware/firmware-main.ino

New HC-SR04 pin definitions
TRIG_PIN 32, ECHO_PIN 33 — no conflict with relay (25), button (27), or RTC I2C (21/22).
New function: readFeedLevelPercent()
Takes median of 3 ultrasonic readings, applies linear interpolation using hopperEmptyCm / hopperFullCm. Returns 0-100%.
Modified: sendTelemetry()
JSON buffer increased from 192 to 256 bytes. Payload now includes feed_level_percent and feed_level_cm on every heartbeat (every 5s).
Modified: pollDashboardFeedCommand()
Parses hopper_full_cm and hopper_empty_cm from backend poll response — calibration updates dynamically, no OTA reflash needed.
